# Ідея проекту — Mise

Апка для домогосподарства, що допомагає **розкласти готування по поличках**: спланувати тиждень
(хто коли що готує), зібрати потрібні інгредієнти, сформувати список покупок і зорієнтувати по
**цінах і знижках** одного конкретного магазину. Назва — від кулінарного _mise en place_
(«усе підготовлене й розкладене по місцях»).

> **Продуктовий принцип:** апка — це **інструмент, а не бюрократія**. Фільтр для будь-якої фічі:
> якщо вона додає облік заради обліку (статуси, погодження, «відмітьте що зробили») — вона **проти**
> принципу. Якщо знімає рішення з голови (що готувати / що купити / скільки коштує) — вона в темі.

Live: https://mise-ua-kaktus-26.vercel.app

## Ролі

У межах **домогосподарства** ролі **симетричні**: кожен учасник одночасно й кухар, і замовник.

- **Кухар** — додає страви, які вміє готувати (свій особистий список). Готує = є власником страви.
- **Замовник** — обирає з каталогу домогосподарства (страви всіх учасників) і ставить їх у спільний календар.
- «Тільки кухар» / «тільки замовник» — **не окремий тип юзера**, а наслідок того, чи людина додала
  страви. Хто не додав жодної — де-факто лише замовник. Окремих рольових прапорців під це немає.
- **Адмін** (`is_admin`) — курує канонічний довідник інгредієнтів, ціни й маппінг на магазин.
  Окремої складної рольової системи немає — лише прапорець.

**Без публічної реєстрації.** Акаунти створюються вручну (2–4 тестових для себе/друзів + демо для
співбесіди). Auth справжній (Supabase), просто публічний sign-up вимкнено → реальна auth/RLS-модель
без відкритих дверей назовні.

## Сутності (таблиці Supabase)

| Таблиця             | Поля (орієнтовно)                                                                                  | Хто редагує          |
| ------------------- | -------------------------------------------------------------------------------------------------- | -------------------- |
| `users`             | Supabase auth, `is_admin`                                                                          | manual / адмін       |
| `households`        | name, invite_code                                                                                  | будь-хто (свій)      |
| `household_members` | user_id, household_id (m2m)                                                                        | будь-хто (свій)      |
| `dishes`            | **owner_id**, household_id, name, photo_url, description, notes, recipe_link                       | лише власник         |
| `ingredients`       | name, **base_unit** (г/мл/шт), **kind** (`weight` \| `packaged`), package_size?, store_product_id? | адмін                |
| `dish_ingredients`  | dish_id, ingredient_id, amount, unit (m2m + дані)                                                  | власник страви       |
| `plan_entries`      | household_id, dish_id, date, created_by (кухар = `dish.owner`)                                     | будь-хто в household |
| `ingredient_prices` | ingredient_id, price, old_price, **is_promo**, store, fetched_at                                   | джоб / адмін         |

Примітки:

- **Модель = «особисті списки» (Варіант 1).** Страва належить автору. Каталог домогосподарства =
  обʼєднання списків усіх. «Обоє вміють пасту» = дві страви; лейбл кухаря в календарі знімає
  неоднозначність. Перетин списків — **не баг**: різні люди роблять різну пасту. Household виправданий
  не дедуплікацією, а **шаром координації** (бачу/замовляю чуже, спільний календар, спільний список покупок).
- **Канонічний довідник інгредієнтів** — серце нормалізації. Кухар при додаванні страви **обирає з
  довідника** (автокомпліт), не друкує вільний текст → «огірок» і «огірки» зводяться в один запис →
  агрегація списку покупок і ціни сходяться. Без цього вони б не агрегувались.
- **Одиниці.** Канонічний інгредієнт має базову одиницю (г / мл / шт). Кількість у `dish_ingredients`
  зводиться до неї. Для MVP — 2–3 базові одиниці, не ускладнювати.
- `photo_url` — реалізація (URL чи Supabase Storage upload) вирішується пізніше.

## Похідні дані (НЕ зберігаються, рахуються при рендері)

- **Вартість страви** = Σ(`amount` зведена до базової одиниці × ціна за базову одиницю).
  **Пропорційна** — для меню/підсвітки (інакше одна страва «зʼїдала б» цілу упаковку, що нечесно).
- **Список покупок** (нетривіальна логіка, ядро апки):
  1. просумувати **сирі потреби** по інгредієнту через усі заплановані страви обраних днів;
  2. звести до базової одиниці;
  3. для **`packaged`** — округлити вгору до цілих упаковок **після агрегації** (3×300 мл молока = 900 мл → 1 пачка 1 л, а не 3);
  4. для **`weight`** — пропорційно, без округлення;
  5. × ціна.
- **Вартість тижня/закупівлі** = Σ по списку покупок (вже з округленням упаковок).
- **Підсвітка «найвигідніше»** = страви, де інгредієнти зараз `is_promo`.

Жодного збереженого `cost`/`shopping_list` поля — single source of truth це первинні дані + ціни.
_(Правило закріплене в `.claude/rules/architecture.md`.)_

## Скоуп даних

- **Один магазин:** Сільпо, конкретна фізична точка (Трускавець). Сільпо віддає ціни й «цінотижики»
  **під обраний магазин/адресу** — тож прив'язка потрібна. Мульти-магазин **свідомо не робимо**.
- **Ціни — шарами, кожен працює сам по собі:**
  - **Шар 0 (MVP):** ручні ціни. Адмін проставляє ціну канонічному інгредієнту. Нуль зовнішніх залежностей, завжди працює.
  - **Шар 1:** фоновий джоб (cron / Edge Function / GitHub Actions) **раз на добу** тягне ціни для довідника, пише в `ingredient_prices`. Апка читає **лише з власної БД**.
  - **Шар 2:** окремо тягне **фід акцій** (`silpo.ua/offers` — вже курований список знижок під магазин) → виставляє `is_promo`.
- **Парсинг — окремий ізольований сервіс.** Реверс недокументованого внутрішнього API Сільпо
  (формально проти ToS, тому бережливо: раз на добу, кеш, без дудосу). Зламається структура — апка
  живе на ручних/останніх цінах. Деталі: `.claude/rules/pricing-service.md`.
- **Маппінг** `канонічний інгредієнт → store_product_id` — руками через адмінку, **один раз**.
  Авто-матчинг назв не чіпаємо (окремий проєкт).

## Фільтри / пошук / сортування

- **Каталог:** за кухарем (учасником), за наявністю промо-інгредієнтів («що зараз вигідно»).
- **Календар:** діапазон тижня, фільтр по кухарю.
- **Список покупок:** мультивибір днів, групування по інгредієнту.

## Флоу

**Dish CRUD** (лише власник) — багатопільна форма: фото, опис для замовника, інгредієнти (вибір із
довідника + кількість/одиниця), нотатки до рецепта, опційний лінк. Гарна практика **RHF + zod**.

**Планування — спільне й асинхронне.** Будь-хто з household кидає страву на день у спільний календар
**у будь-який момент**, без статусів/погоджень/черг. Кухар = власник страви. Видимість (хто кого на
що поставив) — так; гейт-підтвердження — **ні** (це та бюрократія, яку відсікаємо; для пари
вирішується розмовою наживо). Старт без realtime (last-write-wins по незалежних записах + рефреш).

**Список покупок** — обираєш дні → агрегація сирих потреб → округлення упаковок (packaged) → ціна.

> Технічна безпека (хто має право писати) — на боці **Supabase RLS**, не на приховуванні кнопок.
> Базова політика: read у межах свого household; write — лише власні страви; план — write у межах household.
> Деталі: `.claude/rules/supabase.md`.

## Свідомо ЗА бортом MVP

- **Лайки / замовлення як окремі сутності** — їх зʼїв спільний календар (план = система замовлень). Викинуто **назавжди**.
- **Override кухаря** («сьогодні я приготую її страву») — це адміністрування, не допомога. Проти принципу.
- **Pantry** («що вже є вдома», віднімання зі списку) — окрема сутність, v2.
- **Realtime-синк** календаря — v2.
- **Мульти-магазин**, авто-матчинг назв, структуровані кроки рецепта.

## Розширення (фаза 2+, якщо зайде)

- **Шар 1/2 цін** (джоб + фід знижок), якщо не встигло в MVP. Найсильніша історія на співбесіді:
  cron-сервіс, decoupled-архітектура, data-matching.
- **Realtime** спільного календаря — Supabase realtime + optimistic updates (виправдано: двоє реально
  редагують одночасно).
- **Pantry** — віднімання наявного зі списку покупок.
- **Структуровані кроки рецепта** (зараз — лише нотатки + лінк).
- **B2B-візія (наратив, не build-таргет):** white-label **meal-to-basket** движок для мережі —
  «обери страви за привабливою ціною в нашому магазині → в один клік усі інгредієнти в кошик →
  перевір очима → замов». Збільшує середній чек (продаєш не товар, а готову стравову корзину).
  Велике бачення мотивує і продає проєкт на інтервʼю — але **код пишемо під персональний MVP**.
